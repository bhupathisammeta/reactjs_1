# CI/CD Architecture & Workflows

## Overview

This React project includes a comprehensive CI/CD pipeline using GitHub Actions. The pipeline ensures code quality, automated testing, and continuous deployment.

## Workflow Details

### 1. CI/CD Pipeline Workflow
**File**: `.github/workflows/ci-cd.yml`
**Trigger**: Push to `main` or `develop`, Pull Requests

**Steps**:
1. Checkout code
2. Setup Node.js (matrix: 16.x, 18.x)
3. Install dependencies
4. Run linter (if configured)
5. Run tests
6. Build production bundle
7. Upload build artifacts
8. SonarQube quality scan (optional)

**Timeout**: ~15-20 minutes

### 2. Code Quality Workflow
**File**: `.github/workflows/code-quality.yml`
**Trigger**: Push to `main` or `develop`, Pull Requests

**Checks**:
- ESLint (code style)
- npm audit (dependencies)
- Test coverage reports

**Purpose**: Enforce code standards before merge

### 3. Security Workflow
**File**: `.github/workflows/security.yml`
**Trigger**: Weekly schedule + manual push
**Frequency**: Every Sunday 2:00 AM UTC

**Scans**:
- npm audit (known vulnerabilities)
- OWASP Dependency-Check
- Security annotations in code

### 4. Deployment Workflow
**File**: `.github/workflows/deploy.yml`
**Trigger**: Push to `main` branch only
**Manual**: Can be triggered via workflow_dispatch

**Steps**:
1. Build optimized production bundle
2. Deploy to GitHub Pages
3. Send Slack notification
4. Update deployment status

## Environment Setup

### Local Development Environment

```bash
# Install Node.js and npm
node --version  # v16.x or higher
npm --version   # v8.x or higher

# Install dependencies
npm install

# Start dev server
npm start

# Run tests
npm test

# Build for production
npm run build
```

### GitHub Secrets Configuration

Add these to GitHub Settings → Secrets and variables → Actions:

| Secret | Purpose | Example |
|--------|---------|---------|
| `SONAR_HOST_URL` | SonarQube server URL | `https://sonar.example.com` |
| `SONAR_TOKEN` | SonarQube auth token | (Token from SonarQube) |
| `SLACK_WEBHOOK_URL` | Slack notification | `https://hooks.slack.com/...` |
| `GITHUB_TOKEN` | Auto-generated by GitHub | (Auto-provided) |

## Build & Test Strategy

### Test Execution
```
Tests run on:
├── main branch (blocking)
├── develop branch (blocking)
└── Pull Requests (blocking)

Matrix: Node.js 16.x and 18.x
Parallel builds: 2 versions × tests
Total time: ~5-10 minutes per PR
```

### Build Artifacts
- **Location**: GitHub Actions → Artifacts
- **Name**: `build-artifacts-node-{version}`
- **Retention**: 30 days
- **Size**: ~3-5 MB typically

---

## Deployment Strategy

### Environments

```
Development (develop branch)
  └─ Tests pass → Draft releases

Staging (optional)
  └─ Manual promotion step

Production (main branch)
  └─ Auto-deploy to GitHub Pages
  └─ Slack notification sent
```

### Rollback Procedure

If deployment fails:
1. Revert the commit: `git revert <commit-hash>`
2. Push to main branch
3. New workflow automatically deploys reverted version

## Code Quality Gates

### Pre-merge Requirements

```
✓ All status checks pass
✓ Code review approved
✓ Branch up-to-date with main
✓ Tests passing (100% CI pass rate)
✓ No critical security issues
✓ No linting errors
```

### Branch Protection Rules

Recommended settings:
- Require PR reviews: 1+ approvals
- Dismiss stale reviews: On push
- Require status checks: All workflows
- Require branches be up-to-date: Yes
- Allow force pushes: No
- Allow deletions: No

## Common Issues & Solutions

| Issue | Cause | Solution |
|-------|-------|----------|
| Build timeout | Large dependencies | Optimize `package.json` |
| Tests fail in CI but pass locally | Node version mismatch | Check `node --version` |
| Deploy fails | Missing secrets | Add to GitHub Secrets |
| Slack not notifying | Wrong webhook URL | Verify `SLACK_WEBHOOK_URL` |

## Best Practices

1. **Keep main branch protected** - Require PRs for all changes
2. **Write tests first** - TDD approach benefits CI/CD
3. **Keep dependencies updated** - Regular security updates
4. **Monitor logs** - Check Actions tab regularly
5. **Document changes** - Good commit messages
6. **Use feature branches** - Never commit directly to main
7. **Review workflow logs** - Understand why builds fail
